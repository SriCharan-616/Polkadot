## Data Flow Diagrams

### Creating a Proposal

```
User Input (React form)
    ↓
Validation (options count, blocks, thresholds)
    ↓
Call createProposal() on contract
    ↓
Contract Creates Proposal {
    id: proposalCount++,
    status: PENDING_DKG,
    encryptedTally: [(1, 1), (1, 1), ...],
    voteCount: 0
}
    ↓
Emit ProposalCreated event
    ↓
Update UI with new proposal
    ↓
Keyholders listen and respond
```

### DKG Phase

```
ProposalCreated Event
    ↓
[Keyholder 1] Generate random key share s1
    ↓ pk1 = g^s1 mod p
[Keyholder 1] Submit (g^s1, "commitment 1") to contract
    ↓
[Keyholder 2] Generate random key share s2
    ↓ pk2 = g^s2 mod p
[Keyholder 2] Verify pk1 matches previous submissions
    ↓
[Keyholder 2] Submit (g^s1, pk2) to contract
    ↓
[Keyholder 3] Generate random key share s3
    ↓ pk3 = g^s3 mod p
[Keyholder 3] Verify pk matches
    ↓
[Keyholder 3] Submit (g^s1, pk3) to contract
    ↓
[Contract] All 3 submitted ✓
    ↓
[Contract] Set combined publicKey = pk1 * pk2 * pk3 mod p
    ↓
[Contract] Status → ACTIVE
    ↓
Emit VotingStarted event
```

### Voting Phase

```
User selects option and votes
    ↓
[Client] Compute vote weight based on balance
    - Normal: min(balance, 10000)
    - Quadratic: floor(√balance)
    ↓
[Client] Create vote vector: [0, 1, 0, ...] (one-hot at selection)
    ↓
[Client] Multiply by vote weight: [0, 70, 0, ...]
    ↓
[Client] Encrypt each element: E(0), E(70), E(0), ...
    - E(a) = (g^r, g^a * pk^r) for fresh nonce r
    ↓
[Client] Generate ZK proof of eligibility
    - Circuit proves: balance ≥ threshold AND vote weight correct
    - Without revealing: actual balance, option chosen, or identity
    ↓
[Client] Compute nullifier = H([H[privKey, nonce], proposalID])
    ↓
[Client] Submit to contract:
    {
        encryptedVote: [(c1_0,c2_0), ...],
        proof: {a, b, c},
        publicSignals: [pk, threshold, propID, nullifier, ...],
        nullifier
    }
    ↓
[Contract] Verify ZK proof using BN254 precompile
    ↓
[Contract] Check nullifier not already used
    ↓
[Contract] Update encrypted tally homomorphically:
    ∀i: tally[i] = tally[i] ⊙ encryptedVote[i]
       (⊙ is mod multiplication)
    ↓
[Contract] Mark nullifier as used
    ↓
[Contract] Increment vote count
    ↓
Emit VoteCast event
```

### Decryption Phase

```
Voting ends (block.number > endBlock)
    ↓
[Voter/Admin] Call endVoting()
    ↓
[Contract] Check voteCount ≥ minVoterThreshold
    ↓ Success:
[Contract] Status → ENDED
[Contract] Emit VotingEnded event
    ↓
Keyholders watch for VotingEnded event
    ↓
[Keyholder 1] Fetch encrypted tally from contract
    ↓
[Keyholder 1] For each option i:
    partialDecrypt_1[i] = tally[i]^s1 mod p
    ↓
[Keyholder 1] Generate Chaum-Pedersen proof:
    Generate random k
    A = g^k, B = tally[0]^k
    challenge = H(g, tally[0], pk1, pd, A, B)
    response = k - s1 * challenge
    ↓
[Keyholder 1] Submit partialDecryption + proof to contract
    ↓
[Contract] Verify Chaum-Pedersen proof
    ↓
[Contract] Store partial decryption
    ↓
[Keyholder 2] Similarly submit second partial decryption
    ↓
[Contract] Received 2 of 3 ✓
    ↓
[Contract] Call _finalizeResult()
    ↓
[Contract] Lagrange interpolation:
    L0 = (j - i) / (keyholder_0 - keyholder_j)
    L1 = (i - j) / (keyholder_1 - keyholder_i)
    ↓
[Contract] Reconstruct full decryption:
    decrypted = pd_0^L0 * pd_1^L1 mod p
    ↓
[Contract] Compute plaintext:
    For each option i:
    plaintext_i = discrete_log(encTally[i] / decrypted)
    ↓
[Contract] Find winning option
    ↓
[Contract] Status → REVEALED
    ↓
Emit ResultRevealed event
    ↓
Frontend displays results
```

## State Machine

```
┌──────────────┐
│ PENDING_DKG  │  (Waiting for DKG from 3 keyholders)
└──────┬───────┘
       │ (all 3 keyholders submit public keys)
       ↓
┌──────────────┐
│   ACTIVE     │  (Voting window open)
└──────┬───────┘
       │ (block.number > endBlock)
       ↓
   ┌───┴────────────────┐
   │                    │
   ↓                    ↓
┌──────────┐      ┌──────────────┐
│  ENDED   │      │  CANCELLED   │  (< minVoters)
└────┬─────┘      └──────────────┘
     │ (2 of 3 keyholders submit decryptions)
     ↓
┌──────────────┐
│  REVEALED    │  (Results publicly visible)
└──────────────┘
```

## Cryptographic Assumptions

### Security Properties

1. **Vote Privacy**:
   - Semantic security of ElGamal
   - Requires discrete log hardness
   - Ciphertext only reveals no information

2. **Eligibility Privacy**:
   - Soundness of Groth16
   - Zero-knowledge property
   - Cannot forge proof of eligibility

3. **Decryption Integrity**:
   - Completeness of Chaum-Pedersen
   - Soundness (verifier catches cheating)
   - Honest-but-curious keyholder model

4. **Double-Voting Prevention**:
   - Collision resistance of Poseidon
   - Unique nullifier per (voter, proposal) pair
   - Prevents reuse

### Threat Model

**Assumptions:**
- At most 1 of 3 keyholders is corrupted
- Network is asynchronous but reliable
- Contract storage is tamper-proof
- BN254 discrete log is hard
- Hash functions are random oracles

**Threats Handled:**
- Passive keyholders: cannot reveal votes (threshold required)
- Single corrupted keyholder: cannot unilaterally decrypt
- Dishonest voter: caught by ZK proof verification
- Double voting: caught by nullifier check

**Threats NOT Handled:**
- 2+ corrupted keyholders: can conspire to decrypt
- Voter coercion: (general limitation of voting)
- Wallets compromised: outside system scope
- Smart contract bugs: (needs audit)

## Performance Analysis

### Gas Costs (Polkadot PVM)

```
Operation           Gas      Cost (1 Gwei, $1K ETH)
─────────────────────────────────────────────────
createProposal      120,000  $0.12
submitPublicKey     100,000  $0.10
castVote            300,000  $0.30  (includes ZK)
endVoting           50,000   $0.05
submitPartial       250,000  $0.25
─────────────────────────────────────────────────
Per proposal cycle  770,000  $0.77
```

### Proof Generation Time

```
ZK Proof Generation    ~3-5 seconds (client-side)
Chaum-Pedersen Proof   ~0.5 seconds (per keyholder)
Nullifier Computation  ~0.1 seconds
Encryption             ~0.2 seconds per vote
─────────────────────────────────────────────────
Total per vote         ~5 seconds
```

### Storage

```
Per Proposal:
  - Proposal struct:        ~2 KB
  - Encrypted tally:        ~100 bytes × optionCount
  - Nullifier set:          ~32 bytes × voteCount
  - Partial decryptions:    ~512 bytes
  ─────────────────────────────────
  - Total per 1000 votes:   ~35 KB (storage-efficient)
```

### Blockchain Operations

```
Operations per Proposal:
  1. createProposal:     1 transaction (creator)
  2. DKG:                3 transactions (keyholders)
  3. Voting:             voteCount transactions
  4. Decryption:         2-3 transactions (keyholders)
  ─────────────────────────────────────
  Total:                 6 + voteCount transactions
```

## Extensibility

### Possible Extensions

1. **Delegate Voting**
   - Allow delegation of votes
   - Modify circuit: voter can prove delegating

2. **Ranked Choice Voting**
   - Encrypt ranked preferences
   - More complex tally circuit

3. **Secret Ballot with Auditability**
   - Voter can claim their vote later
   - Requires different nullifier scheme

4. **Privacy-Preserving Identity**
   - Ring signatures or mixing
   - Identity != wallet address

5. **Multi-DAO Governance**
   - Shared keyholder infrastructure
   - Multi-proposal batching

6. **Delegation to DAOs**
   - Tokenoholder governance
   - Composability with other protocols
