# Private DAO Voting System on Polkadot PVM - Implementation Guide

This guide walks you through setting up and deploying the complete private voting system.

## Project Structure

```
├── contracts/              # Solidity smart contracts
│   ├── PrivateVoting.sol  # Main voting contract
│   ├── ChaumPedersen.sol  # Zero-knowledge proof verifier
│   └── Verifier.sol       # Groth16 verifier (auto-generated)
├── circuits/              # Circom circuits
│   └── vote.circom        # ZK proof circuit
├── frontend/              # React frontend
│   └── src/
│       ├── pages/         # React pages (Proposals, Vote, Result, etc.)
│       ├── utils/         # Utility functions
│       │   ├── contract.js      # Contract interaction
│       │   ├── elgamal.js       # Encryption utilities
│       │   ├── nullifier.js     # Nullifier generation
│       │   └── zkproof.js       # Proof generation
│       └── components/    # Reusable React components
├── scripts/               # Node.js scripts
│   ├── deploy.js         # Hardhat deployment script
│   └── keyholder.js      # Keyholder server script
├── test/                  # Test files
├── .env.example           # Environment variable template
├── hardhat.config.js      # Hardhat configuration
└── package.json           # Project dependencies
```

## Step 1: Install Dependencies

### Prerequisites
- Node.js 14+ (v16+ recommended)
- npm or yarn
- A Polkadot wallet (for testnet)

### Installation

```bash
# Install root dependencies
npm install

# Install frontend dependencies
cd frontend
npm install
cd ..
```

### Required packages
- `ethers` - Ethereum/Polkadot interaction
- `snarkjs` - ZK proof generation
- `hardhat` - Smart contract development
- `circomlibjs` - Poseidon hash functions
- `recharts` - Charts for frontend

## Step 2: Generate the Groth16 Verifier

The Verifier.sol must be auto-generated by snarkjs. This requires:

1. Compile your Circom circuit to R1CS
2. Create a proving system
3. Export the Solidity verifier

```bash
# Compile circuit (assuming you have circom installed)
circom circuits/vote.circom --r1cs --wasm

# Setup ceremony (using provided pot12 PTAU file)
snarkjs groth16 setup circuits/vote.r1cs circuits/pot12_final.ptau circuits/vote_0000.zkey

# Contribute to ceremony
snarkjs zkey contribute circuits/vote_0000.zkey circuits/vote_final.zkey --name="Contribution" -v

# Export Solidity verifier
snarkjs zkey export solidityverifier circuits/vote_final.zkey contracts/Verifier.sol
```

```bash
npx hardhat compile
```

This will:
- Compile all .sol files in `/contracts`
- Check for syntax errors
- Generate ABIs for frontend use

## Step 5: Deploy to Polkadot PVM

### For Paseo Asset Hub testnet:

```bash
# Set your private key in .env
export DEPLOYER_PRIVATE_KEY=0x...

# Deploy
npx hardhat run scripts/deploy.js --network paseo-asset-hub
```

After deployment:
1. Copy the contract addresses from console output
2. Update .env with the contract address: `REACT_APP_CONTRACT_ADDRESS=0x...`
3. Save deployment addresses to `deployments.json`

## Step 6: Setup Keyholder Servers

Each of the 3 keyholders should run their own server:

```bash
# For keyholder 0
export KEYHOLDER_INDEX=0
export KEYHOLDER_0_PRIVATE_KEY=0x...
export CONTRACT_ADDRESS=0x...
node scripts/keyholder.js
```

The keyholder service will:
- Listen for `ProposalCreated` events
- Submit public key during DKG phase
- Listen for `VotingEnded` events
- Submit partial decryptions

**Security Note**: In production, store key shares in encrypted files. Never commit private keys to git.

## Step 7: Run Frontend

```bash
cd frontend
npm run dev
```

The frontend will be available at `http://localhost:5173` (Vite default)

### Frontend Features:

## Step 8: Testing

### Run Contract Tests

```bash
npx hardhat test
```

### Manual Testing Flow

1. **Create a proposal**
   - Click "Create Proposal"
   - Set description, options, voting mode
   - Set start/end blocks

2. **Wait for DKG (Distributed Key Generation)**
   - Keyholders receive `ProposalCreated` event
   - Each keyholder calls `submitPublicKey()`
   - Once all 3 submit, voting becomes ACTIVE

3. **Cast votes**
   - Go to proposal page
   - Click "Vote"
   - Connect wallet
   - Select option
   - Generate ZK proof
   - Submit vote
   - Vote is encrypted and private

4. **End voting**
   - Click "End Voting" after voting period ends
   - If min voters not met, proposal is CANCELLED
   - Otherwise, status = ENDED

5. **Decrypt results**
   - Keyholders receive `VotingEnded` event
   - Each keyholder calls `submitPartialDecryption()` with Chaum-Pedersen proof
   - Once 2 of 3 submit (THRESHOLD = 2), results are decrypted
   - Status = REVEALED
   - Results are published

## Key Implementation Details

### Homomorphic Encryption
- **ElGamal encryption** preserves the ability to multiply ciphertexts
- `encrypt(a) * encrypt(b) = encrypt(a+b)` in the exponent
- All votes are encrypted with election public key
- Tally is computed as product of all ciphertexts

### Vote Privacy
- **Nullifier** prevents double voting while keeping votes private
- `nullifier = Poseidon(Poseidon(privateKey, DOMAIN_SEPARATOR), proposalId)`
- Voter's identity is not revealed

### ZK Proof
- **Groth16 proof** proves voter eligibility without revealing:
  - Actual token balance
  - Vote choice (encrypted)
  - Private key
- Proof generated client-side, verified on-chain

### Distributed Decryption
- **Shamir secret sharing**: Key is split 2-of-3
- **Chaum-Pedersen proof**: Keyholder proves correct decryption without revealing key share
- Only needed 2 of 3 keys to decrypt (threshold cryptography)

## Troubleshooting

### "Invalid ZK proof"
- Ensure vote.final.zkey is in the correct path
- Verify public signals match circuit definition
- Check that eligibility conditions are met

### "Nullifier already used"
- The same voter already voted on this proposal
- Nullifiers are unique per (voter, proposal) pair

### "Insufficient voters"
- Proposal needs minimum voters before it can end voting
- Set `minVoterThreshold` appropriately

### "Not in ACTIVE state"
- Voting hasn't started yet (waiting for DKG)
- Or voting period has ended
- Or proposal was cancelled

## Security Considerations

### Private Keys
- **Never commit** .env or private keys to git
- Use git-secrets or pre-commit hooks to prevent accidents
- In production: use encrypted storage (AWS KMS, HashiCorp Vault, etc.)

### Contract Verification
- Verify contracts on Blockscout after deployment
- Have external auditors review before mainnet

### Discrete Log Computation
- Current \_finalizeResult uses placeholder
- Production should use off-chain computation + proof submission
- The BabyStepGiantStep algorithm can solve large discrete logs efficiently

### Network Security
- Use HTTPS/WSS for all RPC connections
- Validate contract addresses before interaction
- Implement rate limiting on backend services

## Gas Optimization

### Current Implementation
- castVote: ~200k-300k gas (depends on option count)
- endVoting: ~50k gas
- submitPartialDecryption: ~300k gas

### Optimization Opportunities
- Batch vote processing
- Use storage packing for small values
- Implement EIP-2930 access lists
